## סקירה כללית

המשתמש מתאר במילים חופשיות אימוג'י שהוא רוצה (לדוגמה: *"פרצוף שמח עם עיניים נוצצות"*), והאייג'נט:

1. בודק שהבקשה בכלל שייכת לתחום עיצוב אימוג'ים (Guardrail).
2. מפרק את התיאור ל**רגש**, **מחווה פיזית**, ו**מאפיינים ויזואליים** נפרדים (עיניים, פה, גבות...).
3. בודק מול מאגר RAG של אימוג'ים קיימים אם כבר יש משהו מספיק דומה - ואם כן, חוסך עבודה מיותרת.
4. אם לא - שולף (RAG) עקרונות סגנון רלוונטיים לכל פלטפורמת יעד, ומנסח מפרט עיצובי מלא (צבעים, צורות, קווי מתאר, מאפיינים ייחודיים) בנפרד לכל פלטפורמה.
5. מסביר את ההחלטות שלו, תוך הפניה למקורות הסגנון.
6. שופט את עצמו (LLM-as-judge לפי rubric קבוע) - וחוזר לשפר אם הציון נמוך מדי.

## ארכיטקטורת הגרף
קלט משתמש
│
▼
[Guardrail] ──(מחוץ לתחום)──► Output (הודעת סירוב)
│ (בתחום)
▼
[הבנת רגש] → [זיהוי מחווה] → [פירוק למאפיינים]
│
▼
[בדיקת דמיון קיים] (RAG #1: מאגר אימוג'ים קיימים)
│
▼
Gate: LLM שופט אם המועמדים שנשלפו דומים מספיק
│
├──(דומה מספיק)──► Output (אין צורך באימוג'י חדש)
│
└──(לא דומה)──► [RAG + מפרט לכל פלטפורמה] (RAG #2: מדריכי סגנון)
│
▼
[הסבר החלטות]
│
▼
[Evaluator] ──(ציון < 0.7, עד 3 ניסיונות)──► חזרה ל-RAG+מפרט
│ (ציון ≥ 0.7)
▼
Output

### נודי הגרף

| # | נוד | תפקיד |
|---|---|---|
| 0 | Guardrail | LLM בודק שהבקשה שייכת לתחום עיצוב אימוג'ים; Gate אמיתי שחוסם ומדלג ל-Output |
| 1 | הבנת רגש | LLM מזהה את הרגש המרכזי |
| 2 | זיהוי מחווה | LLM מזהה את הפעולה/מחווה הפיזית, בהקשר הרגש שכבר זוהה |
| 3 | פירוק למאפיינים | LLM עם structured output (Pydantic): עיניים/גבות/פה/רכיבים נוספים בנפרד |
| 4 | בדיקת דמיון קיים | **Retrieve-then-verify**: חיפוש וקטורי (ChromaDB, cosine) שולף 3 מועמדים, ואז LLM נפרד שופט אם אחד מהם *באמת* דומה מספיק (לא מסתמך על סף מרחק קבוע - ראו הסבר בהמשך) |
| 5 | Gate | מותנה בתוצאת נוד 4 - דילוג ל-Output או המשך |
| 6 | RAG + מפרט | לכל פלטפורמת יעד בנפרד: שליפת עקרונות סגנון רלוונטיים (ChromaDB, מסונן לפי platform) + ניסוח מפרט מובנה (Pydantic) |
| 7 | הסבר החלטות | LLM מנמק את הבחירות, תוך הפניה לעקרונות הסגנון שנשלפו |
| — | Evaluator | LLM-as-judge לפי rubric קבוע (`evals/rubric.md`), 4 קריטריונים 1-5; ציון < 0.7 → חזרה לנוד 6 (עד 3 ניסיונות, נמנע לולאה אינסופית) |
| — | Output | מרכיב את הפלט הסופי (מפרט / הודעת סירוב / הודעת "כבר קיים") |

### הרחבות
- **RAG כפול**: מאגר וקטורי נפרד לאימוג'ים קיימים (נוד 4) ולעקרונות סגנון פלטפורמות (נוד 6), שניהם ב-ChromaDB עם embeddings מקומיים (all-MiniLM-L6-v2, ללא עלות).

## התקנה והרצה

### דרישות מוקדמות
- Python 3.10+
- מפתח API מ-[OpenRouter](https://openrouter.ai/keys)
- (אופציונלי) חשבון [LangSmith](https://smith.langchain.com) לניטור

### התקנה

```bash
git clone https://github.com/chagitmi/emoji-spec-agent.git
cd emoji-spec-agent
python -m venv venv
venv\Scripts\activate        # Windows
# source venv/bin/activate   # Mac/Linux
pip install -r requirements.txt
```

### הגדרת סביבה

```bash
copy .env.example .env       # Windows
# cp .env.example .env       # Mac/Linux
```

ערכי את `.env` ומלאי:
- `OPENROUTER_API_KEY` - מ-openrouter.ai/keys
- `LANGCHAIN_API_KEY` (אופציונלי) - מ-smith.langchain.com, להפעלת ניטור

## Evaluations - תוצאות ותובנות

כל Eval שומר תוצאות מפורטות ל-CSV נפרד תחת `evals/results/`.

### 1. השוואת מודלים (`evals/compare_models.py`)

השוואה בין 3 מודלים ב-OpenRouter, מריצה את **הגרף המלא** (לא שלב בודד) על 5 תיאורים קבועים, עם שופט קבוע (`gpt-4o-mini`, `temperature=0`) לכל ההשוואות:

| מודל | אמינות | זמן ממוצע | ציון איכות ממוצע |
|---|---|---|---|
| `openai/gpt-4o-mini` | **100%** | **27.1s** | **0.98** |
| `anthropic/claude-haiku-4.5` | 80% | 50.4s | 0.96 |
| `meta-llama/llama-3.3-70b-instruct:free` | 0% | 175.9s (כולל ניסיונות חוזרים) | — |

**תובנה מרכזית:** המודל בתשלום הזול ביותר היה גם הכי אמין וגם הכי מהיר. המודל היקר יותר (Haiku) נתן איכות דומה אך היה איטי משמעותית ופחות אמין (נתקל פעם אחת במגבלת קרדיט). המודל החינמי היה **לא שמיש בפועל** - נתקל שוב ושוב ב-rate limiting מצד הספק, גם עם retry logic (עד 3 ניסיונות, המתנה של 15 שניות בין ניסיונות) - ממצא לגיטימי לגבי אמינות מודלים חינמיים בסביבת production.

### 2. בדיקת עלויות (`evals/cost_eval.py`)

פירוק עלות **לכל נוד בנפרד**, לא רק סה"כ, על תיאור קבוע:

| נוד | עלות (gpt-4o-mini) |
|---|---|
| Guardrail | $0.00009 |
| הבנת רגש | $0.00006 |
| זיהוי מחווה | $0.0001 |
| פירוק למאפיינים | $0.00025 |
| בדיקת דמיון קיים | $0.00021 |
| **RAG + מפרט** | **$0.0008** |
| הסבר החלטות | $0.0003 |
| Evaluator | $0.00036 |
| **סה"כ** | **~$0.0022** |

**תובנה מרכזית:** נוד "RAG + מפרט" הוא היקר ביותר בפער ניכר (כ-36% מהעלות הכוללת) - הגיוני, כי הוא היחיד שמבצע קריאת LLM **נפרדת לכל פלטפורמת יעד** (3 קריאות בריצה אחת, לעומת קריאה בודדת בכל נוד אחר).

עבור מודל חינמי (`llama-3.3-70b:free`) שנכשל בהרצה בפועל (rate-limit), מוצגת עלות **תיאורטית** מבוססת הערכה למודל בגודל דומה מספק בתשלום, כנדרש.

### 3. מקצועיות ואי-זליגה (`evals/scope_eval.py`)

7 שאלות מחוץ לתחום (יותר מה-5 המינימליים) + 3 שאלות בתוך התחום כ-sanity check:

- **Refusal rate: 100%** (7/7 שאלות מחוץ לתחום נחסמו נכון)
- **False positive rate: 0%** (3/3 שאלות תקינות עברו נכון, לא נחסמו בטעות)

## החלטות טכניות מרכזיות

**Retrieve-then-verify במקום סף מרחק קבוע (נוד 4):** בגרסה ראשונה נעשה שימוש בסף מרחק וקטורי קבוע (`distance < threshold`) כדי להחליט אם אימוג'י קיים "דומה מספיק". התברר שזה לא אמין: חיפוש nearest-neighbor **תמיד** מחזיר את "הכי קרוב שיש", גם כשכלום לא באמת דומה (למשל, בקשה ל"חללית" מול מאגר שמכיל רק פרצופים עדיין תחזיר "הכי פחות רחוק" מביניהם). הפתרון: החיפוש הוקטורי משמש רק לשליפת מועמדים (זול, לצורך שקיפות), ו-LLM נפרד (`temperature=0`) פוסק בפועל אם אחד מהם דומה מספיק סמנטית - דפוס RAG סטנדרטי, נבדק בהצלחה כולל מקרי קיצון.

**`max_tokens` מוגבל במפורש (2000):** ללא הגבלה מפורשת, חלק מהמודלים "מבקשים" את תקרת הטוקנים המקסימלית התיאורטית שלהם (עד 64,000!) כברירת מחדל - מה שגורם ל-API לדרוש יתרה גדולה בהרבה ממה שבאמת נדרש לתשובה בפועל.

**MODEL_JUDGE קבוע בכל ההשוואות:** כנדרש, מודל השופט (`gpt-4o-mini`, `temperature=0`) לעולם לא משתנה בין הרצות ההשוואה ב-Eval 1, כדי לשמור על עקביות ה-rubric.

## מגבלות ידועות

- מודלים חינמיים ב-OpenRouter (`:free`) נתקלים לעיתים קרובות ב-rate limiting מצד הספק, ללא קשר לקוד האייג'נט.
- מאגר האימוג'ים הקיימים (נוד 4) מכיל כרגע 8 דוגמאות בלבד - מספיק להדגמת הארכיטקטורה, אך לא מייצג מאגר אימוג'ים אמיתי.
- מאגר עקרונות הסגנון (נוד 6) מכסה כרגע 3 פלטפורמות (Apple/Google/Samsung).

## Roadmap עתידי

- **7+ פלטפורמות**: הרחבת מאגר עקרונות הסגנון, כולל שימוש ב-LLM עם יכולת ראייה (vision) לתמצת סגנון אוטומטית מתוך דאטה-סט תמונות אמיתי, במקום כתיבה ידנית.
- **קלט תמונה**: אפשרות לספק תמונה (המתארת ישירות את האימוג'י הרצוי) במקום/בנוסף לתיאור טקסטואלי.
- **Code-Exec ל-SVG**: המרת המפרט הטקסטואלי לקובץ SVG בסיסי בפועל, עם אישור HITL לפני יצירת קובץ.
- **שילוב מודל סיווג פלטפורמה** (מפרויקט ML קודם) בתוך ה-Evaluator, בכפוף לבדיקת היתכנות.
